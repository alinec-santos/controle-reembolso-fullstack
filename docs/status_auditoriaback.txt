9. ERROS E INCONSISTÊNCIAS ENCONTRADAS
--------------------------------------------------------------------------------

9.1 ERRO DE CÓDIGO — Importação duplicada em category.routes.ts
    Arquivo: backend/src/routes/category.routes.ts (linhas 8 e 9)
    Problema: `validateParams` está importado duas vezes na mesma linha.
      import { validateParams } from "../middlewares/validate-params.middleware"
      import { validateParams } from "../middlewares/validate-params.middleware"
    Impacto: Erro de compilação TypeScript. Pode não impedir execução, mas é
             code smell grave e pode causar conflito em builds.
    Solução: Remover a linha duplicada.

9.2 INCONSISTÊNCIA — show.controller.ts acessa `history.updatedAt` inexistente
    Arquivo: backend/src/modules/reimbursement/controllers/show.controller.ts (linha 75)
    Problema: O código tenta formatar `history.updatedAt`, mas a entidade
             `RequestHistory` no schema.prisma NÃO possui campo `updatedAt`.
    Impacto: Erro em tempo de execução (TypeScript pode capturar, mas é runtime).
    Solução: Remover `updatedAt: formatDateTime(history.updatedAt)` do map.

9.3 NOTA — Criação de usuário exige autenticação ADMIN
    Arquivo: backend/src/routes/user.routes.ts
    Situação: POST /users requer roleMiddleware([UserRole.ADMIN]), ou seja,
              somente ADMIN pode criar novos usuários.
    Observação: A especificação lista "Cadastro de usuário" como funcionalidade
                obrigatória, mas não especifica explicitamente quem pode criar.
                Se a intenção é que qualquer pessoa possa se registrar (como
                em sistemas SaaS), esta rota deveria ser pública. Se a
                intenção é que apenas ADMIN gerencie usuários, está correto.
                O frontend trata isso como funcionalidade de gerenciamento do Admin.
                Decisão documentada implicitamente. Recomenda-se explicitar no README.

9.4 NOTA — Cancelamento de solicitação ENVIADA está ativo
    Arquivo: backend/src/modules/reimbursement/controllers/cancel.controller.ts
    Situação: O sistema permite cancelar solicitações tanto em RASCUNHO quanto
              em ENVIADO. A especificação diz que cancelar ENVIADO é "opcional
              se o candidato documentar a decisão".
    Impacto: Funcionalidade extra sem documentação explícita da decisão.
    Recomendação: Adicionar comentário/README justificando esta escolha.

9.5 NOTA — Middleware de validação de query params presente mas schema básico
    Arquivo: backend/src/schemas/query.schema.ts (inferido)
    Situação: Existe um `listRequestsQuerySchema` sendo usado na rota GET
              /reimbursements, porém não foi possível confirmar a implementação
              completa de filtros. A lista retorna todos os dados filtrados
              apenas por perfil, sem suporte a query params de filtro no controller.
    Impacto: Validação de query está no middleware mas pode não ser usada no controller.

9.6 OBSERVAÇÃO — Testes backend não cobrem todos os fluxos negativos
    Situação: Os testes cobrem os fluxos principais (criar, enviar, aprovar,
              rejeitar, pagar, listar histórico), mas não cobrem:
              - Rejeição sem justificativa (validação Zod)
              - Tentativa de pagar uma solicitação ENVIADA (transição inválida)
              - Tentativa de colaborador criar solicitação com categoria inativa
              - Acesso de colaborador a solicitação de outro colaborador
    Impacto: Cobertura de testes parcial. A spec exige "testes de integração
             para rotas principais, autenticação, validações e regras de negócio".
